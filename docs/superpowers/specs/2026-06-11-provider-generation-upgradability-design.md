# Provider Generation & Upgradability — Design

- **Date:** 2026-06-11
- **Status:** Approved (design); ready for implementation planning
- **Scope:** Unified design covering (a) clean, reliable, idempotent generation and
  (b) safe upgrade of core components in already-generated providers.

## 1. Problem

Two connected problems with how `xp-provider-gen` produces and maintains providers:

1. **Generation leaves a dirty, "not fresh" tree.** `automation/pipeline.go` commits
   *before* running the file-producing steps. The init pipeline runs
   `git init → git commit → git submodule add build → make submodules → go mod tidy →
   make generate → make reviewable` — so `.gitmodules`, `build/`, rewritten
   `go.mod`/`go.sum`, all `zz_generated.*`, CRDs, and examples are produced **after** the
   commit and left uncommitted. The "Initial commit" captures only the bare skeleton. The
   `create api` pipeline (`git commit → make generate`) has the same defect. Every command
   leaves a dirty working tree. Worse, all make/tidy steps are marked **non-required**, so a
   command reports success even when generation actually failed.

2. **No upgrade path for core components.** Once generated, the controller wiring,
   registration files, build wiring, ProviderConfig plumbing, and pinned dependency versions
   are static copies. The tool cannot re-apply updated core templates to an existing provider,
   there is no marker separating tool-owned from user-owned code, and no record of which
   generator version produced a file. Updating the generator cannot safely propagate to
   existing providers (today the README says breaking changes require full regeneration).

A third, underlying issue feeds both: the registration files are kept current by **parsing
and merging** (`file_parser.go`, `api_registration_updater.go`, `template_updater.go`) — a
fragile regex/section-marker subsystem that is the bulk of the "too many moving pieces"
maintainability concern.

## 2. Decisions (keystone choices)

| Decision | Choice |
|----------|--------|
| Scope | Unified: clean generation **and** upgradability in one design |
| Separation model | **File-level ownership**; PROJECT manifest is the source of truth |
| Upgrade mechanism | New **`update`** command; safety via a `DO NOT EDIT` header; generator version stamped in PROJECT; changes reviewed as a git diff |
| Git behavior | **Generate-then-commit** for `init`/`create api` (one clean commit, clean tree); `update` leaves a diff (no auto-commit); all steps **required** |

## 3. Design

### 3.1 Ownership contract

Every file falls into exactly one bucket, decided by the presence/origin of a
`DO NOT EDIT` header:

| Bucket | Files | Header | On `update` |
|--------|-------|--------|-------------|
| **Tool-owned** | `setup.go`¹, all `register.go`, `internal/controller/config/config.go`, `cmd/provider/main.go`, `hack/boilerplate.go.txt`, `Makefile`, `Dockerfile`, `package/crossplane.yaml`¹ | `// Code generated by xp-provider-gen. DO NOT EDIT.` | Always overwritten |
| **Codegen-owned** | `zz_generated.*`, `package/crds/*` | controller-gen's own `DO NOT EDIT` | Regenerated by `make generate` |
| **User-owned** | `internal/controller/<kind>/controller.go`, `apis/<grp>/<ver>/<kind>_types.go` | none | Never touched |

¹ `setup.go` and the structural fields of `crossplane.yaml` are **open decisions** (§7).

**The overwrite gate** — the single rule the tool applies before writing any file:

```
if target has the xp-provider-gen DO NOT EDIT header → overwrite   (tool-owned)
elif target does not exist                            → write      (seed user-owned)
else (exists, no header)                              → skip        (never touch)
```

`controller.go` and the user-added type fields have no header → they always hit `skip`.
`<kind>_types.go` is seeded once as a skeleton; the user adds fields; controller-gen turns
those into the headered `zz_generated.deepcopy.go`.

### 3.2 Deterministic generation core

All commands share one core: render every tool-owned file from templates + the PROJECT
manifest, seed user-owned files only if absent, run codegen, leave a clean tree.

**Registration files become deterministic.** Because PROJECT already lists every resource,
`register.go` files are regenerated wholesale from that list rather than parsed-and-merged.
The parse-and-merge subsystem (`file_parser.go`, `api_registration_updater.go`,
`template_updater.go`) is **removed**.

### 3.3 Command flows

**`init`**
```
validate flags → write PROJECT manifest → render tool-owned files
  → seed user-owned files (ProviderConfig skeleton)
  → git init → make submodules → go mod tidy → make generate → make reviewable
  → ONE commit  (clean tree)
```

**`create api`**
```
validate resource → add resource to PROJECT manifest
  → regenerate ALL tool-owned files from manifest (register.go deterministic)
  → seed user-owned files for the new kind (controller.go stub, <kind>_types.go skeleton) IF ABSENT
  → make generate → make reviewable
  → ONE commit  (clean tree)
```

**`update`** (new)
```
read PROJECT manifest + stamped generator version + dependency manifest (§3.4)
  → re-render every tool-owned file (header present → overwrite, with drift check §3.5)
  → apply dependency versions via `go get` (does NOT overwrite go.mod)
  → user-owned files skipped entirely
  → go mod tidy → make generate → make reviewable
  → STOP — no commit; user reviews `git diff` and commits
  → re-stamp generator version + provenance in PROJECT
```

`create api` and `update` use the **same regeneration core**, differing only in whether the
manifest gains a resource first — so `update` cannot drift from `create`.

### 3.4 Dependency version manifest

A single config file in the generator is the source of truth for every dependency the
generated `go.mod` declares:

```yaml
# pkg/versions/dependencies.yaml
dependencies:
  github.com/crossplane/crossplane-runtime/v2: v2.0.0
  sigs.k8s.io/controller-runtime:               v0.21.0
  k8s.io/apimachinery:                          v0.33.3
  k8s.io/client-go:                             v0.33.3
  sigs.k8s.io/controller-tools:                 <version>   # controller-gen
  github.com/crossplane/crossplane-tools:       <version>   # angryjet
  # ...every direct require/tool in go.mod.tmpl
```

This drives three things:

1. **Rendering** — `go.mod.tmpl` and the `tool` directives render their versions from this
   manifest; no version string is hardcoded in a template (DRY).
2. **Upstream tracking** — a Renovate **custom (regex) manager** in the generator watches this
   file, so every listed dependency (crossplane-runtime, all `k8s.io/*`, controller-runtime,
   the codegen tools) gets its own upstream-bump PR against the generator, gated by CI + e2e
   before it ships. This closes today's gap, where these versions are literal strings in
   `go.mod.tmpl` and invisible to Renovate.
3. **Provider upgrades** — `update` applies each entry via `go get <dep>@<version>`, leaving
   the user's own `require` lines untouched, then regenerates the tool-owned wiring that
   depends on the new versions.

**Scope guardrail:** the manifest tracks only the **direct** dependencies the generated
`go.mod` declares; Renovate + `go mod tidy` handle transitive ones.

**Provenance:** PROJECT records `xpProviderGenVersion` as the canonical marker; the exact
dependency set is whatever that generator version's `dependencies.yaml` pins. crossplane-runtime
is the primary motivating dependency — its bump and the matching `setup.go` wiring change move
together, in lockstep, via one `update`.

### 3.5 Safety guarantees

- **Overwrite gate** (§3.1) makes "never touch user code" structural, not best-effort.
- **Drift detection:** before overwriting a headered file, the tool checks whether it still
  matches what it would regenerate. Matches → overwrite silently. Differs → the file was
  hand-edited despite the header; the tool **refuses to overwrite, reports it as drift, and
  aborts** with guidance. No surprise clobbering.
- **Version stamping & staleness:** PROJECT records the generator version that last wrote the
  tree. `update` reports from→to; `init`/`create api` warn if run with an older tool than last
  touched the project; a future `update --check` dry-run can report what would change.
- **Review surface:** `update` leaves everything uncommitted, so `git diff`/`git status` is the
  review tool. Any user-owned file appearing in the diff is an immediately visible contract bug.

The worst case is an aborted `update` with a clear drift report — never silent loss of logic.

### 3.6 Reliability

- All pipeline steps become **required**; a failure aborts with a clear error.
- Each command **asserts clean state** at the end: `init`/`create api` verify a clean tree
  after their commit; `update` verifies only tool-owned/codegen files appear in `git status`.
- The parse-and-merge subsystem is **deleted** (see §3.2).

## 4. Testing strategy

- **Unit:** overwrite gate (header→overwrite, absent→seed, headerless→skip); drift detection
  (matches→overwrite, differs→abort); manifest→`go.mod` rendering; deterministic `register.go`
  generation.
- **e2e (extend `scripts/e2e-test.sh`):**
  - After init + create api, assert **`git status` is clean** (regression test for problem #1).
  - Add an **`update` scenario:** hand-edit a `controller.go`, run `update`, assert (a) the edit
    survives, (b) tool-owned files refreshed, (c) a deliberately hand-edited tool-owned file
    triggers a drift abort.

## 5. Migration (existing providers)

Existing providers lack headers and the version stamp. Add a one-shot **`update --adopt`** that
stamps `xpProviderGenVersion` + dependency provenance into PROJECT and writes headers onto the
files it recognizes as tool-owned (by path), after which normal `update` works. Reviewed as a
normal git diff. Adoption is the natural moment to resolve the §7 open decisions.

## 6. Documentation review & consolidation

This design changes the architecture (new generation core, removed parse-and-merge subsystem,
new `update` command, dependency manifest, version stamping), so the existing docs will go
stale. As part of this work, review and consolidate everything under `docs/` for accuracy and
non-duplication:

- **`docs/architecture.md`** — replace the parse-and-merge/updater description with the
  deterministic generation core; document the ownership contract, the `update` command, and the
  dependency manifest.
- **`docs/development.md`** — add the `update` workflow and the dependency-manifest bump flow;
  reconcile the "working with templates" section with file-level ownership.
- **`docs/testing.md`** — document the clean-tree assertion and the `update`/drift e2e scenario.
- **`README.md`** — remove the "existing projects need regeneration" caveat once `update` lands;
  ensure the commands section lists `update`.
- **`.github/WORKFLOWS.md`** — note the Renovate custom manager for the dependency manifest.
- **Cross-cutting:** verify no doc duplicates another (DRY); each file states its purpose once
  and links rather than repeats; all version numbers and command names match reality.

Deliverable: a short consistency pass with the docs updated in the same change set as the code,
so documentation never lags the implementation.

## 7. Open decisions (not blockers)

1. **`setup.go` ownership.** It is mechanical crossplane-runtime wiring with no business logic,
   which makes it the prime "core component" to upgrade — argues for tool-owned. The only
   counter is rare local tuning (poll interval, feature toggles). Default: **tool-owned**;
   confirm during adoption.
2. **`crossplane.yaml` structural fields.** Mixed-ownership: `apiVersion`/`kind`/`capabilities`/
   `controller.image` track the Crossplane packaging model (tool concern), while
   `maintainer`/`description`/`readme` are user prose. Options: split the file, managed regions,
   or seed-once-never-touch for the prose with tool-owned structural fields. To be resolved.

## 8. Out of scope (YAGNI)

- Tracking the full transitive dependency graph (only direct deps are manifested).
- Marker-based in-file regions and config-driven (upjet-style) regeneration (rejected in favor
  of file-level ownership).
- Automatic conflict *resolution* on drift (the tool reports and aborts; the human reconciles).
